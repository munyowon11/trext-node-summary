# trext-node-summary

Read-only Slurm node status summary.

## Purpose

`trext-node-summary` is a small utility for summarizing Slurm node and partition status from text output.

The goal is to make Slurm status output easier to review without changing cluster state.

## Scope

This repository focuses on:

- parsing sample `sinfo` output
- summarizing node states
- counting common states such as `idle`, `alloc`, `down`, `drain`, and `mix`
- producing a simple one-page status summary
- keeping all behavior read-only

## Non-goals

This project does not:

- control Slurm
- modify Slurm configuration
- drain, resume, reboot, or modify nodes
- send commands to cluster infrastructure
- connect to external services
- upload or transmit cluster data
- process private operational data
- claim root cause, savings, reliability improvement, or production readiness

## Safety Boundary

This repository is read-only by design.

Any output generated by this tool should be treated as a review aid, not as an operational decision or automated control signal.

Human review is required before taking any operational action.

```bash

```

This command reads the sample input file only.
It does not connect to a live Slurm cluster.

## Public safety boundary

This tool is intended to be read-only.

## Planned Input

Initial input should be sample text similar to:

```text
PARTITION AVAIL  TIMELIMIT  NODES  STATE NODELIST
gpu          up   infinite      2  alloc gpu-[001-002]
gpu          up   infinite      1   idle gpu-003
gpu          up   infinite      1  drain gpu-004
cpu          up   infinite      3   idle cpu-[001-003]
```

## Planned Output

Initial output should be a short summary such as:

```text
Slurm Node Status Summary

Total nodes: 7
Idle: 4
Allocated: 2
Drain: 1
Down: 0
Mixed: 0

Review notes:
- 1 node is marked drain.
- Check drain reason before making any operational decision.
- This report is read-only and does not determine root cause.
```

## One-page Summary Format

The one-page summary should include:

- cluster status overview
- node state counts
- partitions requiring review
- possible time windows requiring human review, if available in sample input
- questions to check before diagnosing incidents
- clear limitations
